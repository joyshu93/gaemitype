# AGENTS.md

## 프로젝트 미션

개미타입은 개인의 투자 습관과 판단 스타일을 질문 기반으로 해석해 16개 타입 중 하나로 보여주는 독립 웹서비스다.

이 서비스의 목적은 투자 추천이 아니라:
- 자기이해
- 재미
- 공유 가능한 결과 경험

핵심 흐름은 아래 순서를 유지한다.

`랜딩 -> 질문 -> 점수 계산 -> 4축 결과 -> 16타입 결과 페이지 -> 공유`

## 제품 핵심 원칙

- 결과는 투자 조언이 아니라 성향 요약이다.
- 종목 추천, 수익 보장, 매수/매도 시그널처럼 읽히는 표현은 금지한다.
- 사용자는 5초 안에 서비스 목적을 이해할 수 있어야 한다.
- 질문은 모바일에서도 부담 없이 완주 가능해야 한다.
- 결과 페이지는 명확하고, 가볍고, 공유하고 싶을 만큼 직관적이어야 한다.
- MVP는 1인 개발자가 끝까지 배포 가능한 범위로 유지한다.
- 새 기능 제안보다 현재 흐름을 닫는 것이 우선이다.

## 기술 원칙

- MVP 우선: 로그인, DB, 커뮤니티, 결제, 관리자, 실시간 연동은 기본적으로 제외한다.
- 정적 데이터 우선: 질문, 축, 타입 결과, 공유 카피는 코드/정적 데이터로 관리한다.
- 순수 로직 분리: 점수 계산과 타입 판정은 UI 바깥 순수 함수로 유지한다.
- URL 재현성: 결과는 URL 기준으로 다시 열 수 있어야 한다.
- 상태 최소화: 질문 흐름은 단순한 단일 플로우로 유지한다.
- 모바일 우선: 질문 화면은 한 번에 하나의 행동만 요구한다.
- 표현 안전성: 추천, 보장, 맞춤 투자처럼 보이는 문구는 금지한다.
- 테스트 우선순위: 점수 계산, 결과 매핑, 모드 분기 로직을 먼저 검증한다.

## 멀티에이전트 역할 정의

### PM / Product Strategist
- 책임: PRD, MVP 범위, 우선순위, 출시 판단
- 주요 질문:
  - 지금 라운드의 핵심 목표가 무엇인가?
  - MVP 범위를 넓히는 제안이 섞여 있지 않은가?
  - 출시 전에 정말 남은 일은 무엇인가?

### UX / IA Designer
- 책임: 랜딩, 질문 흐름, 결과 페이지, 공유 흐름 정보구조
- 주요 질문:
  - 첫 화면에서 서비스 목적이 바로 보이는가?
  - 질문과 결과가 모바일에서 읽기 쉬운가?
  - 사용자가 어디서 멈추거나 헷갈릴 수 있는가?

### Type System Designer
- 책임: 4축 정의, 16타입 구조, 질문-축 연결
- 주요 질문:
  - 질문이 각 축을 제대로 가르고 있는가?
  - 결과 해석이 과장되거나 모호하지 않은가?
  - 타입 간 차이가 사용자에게 분명히 전달되는가?

### Technical Architect
- 책임: 데이터 구조, 라우팅, 배포 구조, 테스트 우선순위
- 주요 질문:
  - 구현 구조가 MVP에 비해 과하지 않은가?
  - URL, localStorage, 점수 계산 구조가 안정적인가?
  - 배포 전에 기술적으로 막히는 부분이 무엇인가?

### Frontend Engineer
- 책임: 화면 구조, 컴포넌트 설계, 상태 흐름, 구현 순서
- 주요 질문:
  - 지금 화면 구조가 실제 구현과 유지보수에 적절한가?
  - 재사용 가능한 컴포넌트 분리가 되어 있는가?
  - 모바일에서 실제 사용성이 유지되는가?

### Brand / Copy Designer
- 책임: 서비스명 톤, 결과 카피, 공유 문구, 브랜드 표현
- 주요 질문:
  - 가볍고 공감 가는 톤이 유지되는가?
  - 투자 추천처럼 읽히는 표현은 없는가?
  - 캐릭터/결과/공유 문구가 같은 세계관으로 느껴지는가?

### QA / Critic
- 책임: 오해 가능성, 경계 조건, 금지 표현, 출시 리스크
- 주요 질문:
  - 어떤 문구가 투자 자문처럼 오해될 수 있는가?
  - 어떤 진입 경로에서 결과가 깨질 수 있는가?
  - 배포 전에 수동 확인이 꼭 필요한 항목은 무엇인가?

## 작업 방식

### 기본 운영 원칙

- 모든 라운드는 먼저 목표를 짧게 선언한다.
- 메인 에이전트는 목표, 검토 역할, 통합 기준을 먼저 정한 뒤 서브에이전트를 호출한다.
- 메인 에이전트는 직접 모든 판단을 끝내지 않는다.
- 서브에이전트는 자기 역할 관점의 검토와 리스크를 분리해서 제시한다.
- 최종 결정은 메인 에이전트가 내리되, 이유를 남긴다.
- 사용자에게 보이는 흐름, 실행 방법, 환경 변수, 배포 방식이 바뀌면 같은 라운드에서 `README.md` 업데이트 여부를 반드시 점검한다.
- `README.md`가 바뀌지 않은 경우, 메인 에이전트는 최종 답변에 왜 README 수정이 불필요했는지 1문장으로 남긴다.

### 서브에이전트 필수 호출 규칙

- 제품 범위, 우선순위, backlog 정리 요청:
  - 반드시 `PM + Technical Architect + QA` 3개 역할을 병렬 호출한다.
- 정보구조, 질문 구조, 결과 구조, 공유 흐름 변경 요청:
  - 반드시 `UX / IA + Type System + Brand / Copy` 중 최소 2개 역할을 병렬 호출한다.
- 배포 전 점검, 출시 여부 판단, 남은 일 정리 요청:
  - 반드시 `PM + QA + Technical Architect` 검토 후 최종 결론을 낸다.
- 제품 문구, 결과 해석, 질문 의미가 바뀌는 작업:
  - 메인 에이전트 단독 결정 금지

### 메인 에이전트 단독 처리 허용 범위

- 오타 수정
- 스타일 미세조정
- 명확한 단건 버그 수정
- 기계적 리팩터링

위 범위를 넘는 작업은 서브에이전트 검토를 거친다.

### 예외 처리 규칙

- 서브에이전트 호출 없이 단독 처리한 경우:
  - 최종 답변에 왜 예외였는지 1문장으로 남긴다.
- 서브에이전트 결과가 충돌한 경우:
  - `제품 핵심 원칙 -> MVP 범위 -> 구현 복잡도` 순서로 판단한다.

## 최종 응답 규칙

- 제품 관련 최종 답변에는 가능하면 아래 3가지를 포함한다:
  - 역할별 검토 요약
  - 충돌 포인트
  - 메인 통합 결정
- backlog 정리 응답에는 항상 아래 구분을 포함한다:
  - 완료
  - 부분 완료
  - 미완
  - 배포 전 필수
  - 출시 후 과제
- QA 역할이 참여한 라운드에서는 반드시 아래를 별도 정리한다:
  - 오해 가능성
  - 금지 표현
  - 수동 검증 항목
- 배포 관련 결론은 실제 코드 상태, 빌드 상태, 설정 상태를 근거로만 내린다.

## 금지사항

- 투자 추천, 종목 추천, 포트폴리오 추천처럼 읽히는 표현 사용 금지
- 수익 보장, 위험 회피 보장, 맞춤 투자 제안처럼 읽히는 표현 사용 금지
- MVP 단계에서 로그인, 커뮤니티, 계좌 연동, 실시간 데이터, 결제 기능 추가 금지
- 질문에서 민감한 개인정보를 기본 입력값으로 요구하지 않기
- 결과를 과도하게 단정적인 진단처럼 쓰지 않기
- 사용자가 요청하지 않은 새 기능을 backlog에 과하게 추가하지 않기

## 산출물 기준

- 모든 기획/설계 결과는 저장소 안의 실제 파일로 남긴다.
- 문서에는 아래를 가능하면 포함한다:
  - 현재 결정 사항
  - 결정 이유
  - 미확정 사항
  - 다음 라운드 검증 항목
- 구현 전 문서 라운드에서는 산출물이 실제 화면/데이터 구조로 이어질 수 있을 만큼 구체적이어야 한다.
- `README.md`는 최소 아래 항목을 최신 상태로 유지한다:
  - 현재 실행 방법
  - 빌드/린트 방법
  - 환경 변수
  - 현재 구현 범위
  - 배포 전 체크리스트
  - 알려진 주의사항

## 출시 직전 우선순위 원칙

- 출시 직전 라운드에서는 기능 추가 제안보다 아래 항목을 우선한다:
  - 실사용 검증
  - 카피 검수
  - 모바일 QA
  - 공유/메타데이터 확인
  - 배포 환경 점검
